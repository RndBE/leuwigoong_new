Laporan Investigasi — Kontrol Pintu Air AWGC Leuwigoong

Temuan: (1) Pintu Intake tidak dapat dibuka  •  (2) Command Sirine / EWS
Sistem: AWGC (Automatic Water Gate Control) — Leuwigoong / Copong, Garut  |  Tanggal: 4 Juni 2026  |  Sifat: Investigasi read-only (tidak ada perintah kontrol yang dikirim, tidak ada file diubah)
Ringkasan Eksekutif. Klien melaporkan pintu intake tidak bisa dibuka via web. Setelah eliminasi (topik MQTT sama untuk kedua logger; gating R/S/T & status_controller LOLOS; logger 10349 online), penyebab paling mungkin adalah perbedaan struktur data pada payload MQTT: untuk intake (logger 10349) data dikirim sebagai array datar, sedangkan untuk floodway (10350) sebagai array-of-arrays — padahal keduanya memakai "setting": "multi_gcm". Perlu konfirmasi tim hardware mengenai bentuk yang diharapkan firmware multi_gcm. Sebagai temuan tambahan, command sirine (EWS / ews_onoff) ditemukan, namun dengan beberapa kejanggalan (lihat Bagian 2).

1.2 Penyebab paling mungkin: bentuk data berbeda

Pada application/controllers/Kontrol.php baris 98–102:

if($this->session->userdata('idlogger') == '10350'){
    $items[] = [$gcm, (int)$v['elev'], 1, 0];   // 10350 → array-of-arrays (NESTED)
}else{
    $items   = [$gcm, (int)$v['elev'], 1, 0];   // 10349 → array datar (FLAT)
}

Kedua jalur dikirim dengan "setting": "multi_gcm" (baris 134) ke topik yang sama. Perbedaannya hanya pada bentuk data:

LoggerPayload data yang dikirimDugaan reaksi firmware multi_gcm
10350
Floodway		 [ ["gcm1", elev, 1, 0] ]
(array-of-arrays)		 Iterasi → tiap elemen berupa array → perintah valid → pintu bergerak ✓
10349
Intake		 [ "gcm5", elev, 1, 0 ]
(array datar)		 Iterasi → tiap elemen = string/angka, bukan array → gagal parse / di-skip → pintu diam ✗
Hipotesis teknis. Karena multi_gcm berarti "daftar perintah pintu", firmware kemungkinan me-loop isi data dan mengharap tiap elemen berbentuk [gcm, elevasi, status, 0]. Payload intake yang datar membuat firmware mengiterasi elemen bertipe salah → perintah tidak terbentuk → command "terkirim" ke broker tetapi tidak dieksekusi device. Inilah yang konsisten dengan gejala "pintu intake tidak bisa dibuka, tanpa error di web".
Pertanyaan untuk tim hardware (mohon dikonfirmasi):
  • Untuk "setting": "multi_gcm", apakah firmware mengharap data berupa array-of-arrays [[gcm,elev,status,0]]?
  • Jika menerima array datar ["gcm5",elev,1,0], apakah firmware mengabaikannya / error / crash?
  • Apakah ada log di sisi logger 10349 yang mencatat pesan diterima namun gagal di-parse?

1.3 Usulan perbaikan (menunggu konfirmasi hardware)

Jika firmware mengharap array-of-arrays, samakan kedua cabang menjadi nested untuk semua logger:

// Kontrol.php — ganti baris 98–102 menjadi satu baris (hapus if/else):
$items[] = [$gcm, (int)$v['elev'], 1, 0];

Catatan: single-gate per sesi tetap by-design — perbaikan ini hanya membetulkan bentuk payload, bukan menambah multi-pintu.

BAGIAN 2 — Command Sirine (EWS)

JAWABAN Ya, ada command untuk menyalakan sirine, berupa Early Warning System dengan "setting": "ews_onoff". Hanya ditemukan di application/controllers/Api.php (fungsi lanjut_kontrol() baris ~1459 dan lanjut_kontrol2() baris ~1639).

$ews = [
    "set_10350" => [               // <- selalu hardcoded 10350
        "command" => "set",
        "setting" => "ews_onoff",
        "data"    => ["1"]            // <- hanya "1" (NYALA), tidak ada "0"
    ]
];
$mqtt->publish('AWGC_Garut_Copong', json_encode($ews), 0);

2.1 Kejanggalan yang ditemukan

#TemuanDampak
1 Key EWS selalu di-hardcode "set_10350" di kedua fungsi (Api.php:1460 & 1640), walau yang dikontrol logger 10349. Sirine yang dinyalakan selalu milik logger 10350 — tidak bisa menyalakan sirine spesifik 10349.
2 Hanya ada perintah NYALA ("data":["1"]). Tidak ditemukan command mematikan ("0") di seluruh kode. Mematikan sirine bergantung pada logika di firmware (mis. timer), bukan dikontrol web.
3 Pemicu sirine tidak konsisten: di lanjut_kontrol() sirine dipicu saat id_logger=='10350' (baris 1470); di lanjut_kontrol2() saat id_logger=='10349' (baris 1650). Perilaku sirine berbeda tergantung endpoint API mana yang dipakai aplikasi.
4 Di jalur kontrol web utama (Kontrol.php) tidak ada command sirine sama sekali. Operasi pintu lewat web utama tidak memicu peringatan sirine.
Pertanyaan untuk tim hardware (sirine):
  • Apakah sirine memang hanya satu unit (milik 10350) untuk seluruh lokasi, atau tiap logger punya sirine sendiri?
  • Bagaimana sirine dimatikan — apakah firmware mematikan otomatis setelah durasi tertentu, atau perlu command ews_onoff "0"?
  • Apakah perilaku sirine yang berbeda antar endpoint (lihat temuan #3) memang disengaja?
